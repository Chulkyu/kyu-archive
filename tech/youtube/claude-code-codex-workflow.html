🚑
구조 기능

Codex Rescue

막힌 버그에서 슬래시 커맨드 하나. Codex가 백그라운드로 분석하는 동안 다른 작업 계속 가능.

6분 35초 만에 엣지 케이스 버그 4개 발견 사례

💰
비용 최적화

Opus + Sonnet 역할 분리

플래닝은 Opus(비쌈), 구현은 Sonnet(저렴). 설계 10~20% + 구현 80% 구조로 비용 절감.

$1.50 → $0.48 / 10만 토큰 (약 3분의 1)

🔍
설계 검증

Codex Adversarial Review

플랜 단계에서 설계를 비판적으로 검토. 코드 짜기 전 수정이라 비용이 거의 안 듦.

3라운드 돌려서 설계 문제 14개 발견 사례

🔄
권장 워크플로우

4단계 통합 워크플로우

🔎 실제 사례: Claude가 놓친 버그 3가지

어떤 분이 Claude Code로 MCB 브릿지 코드를 만들었는데, Codex로 리뷰시켰더니 발견된 문제들:

🕳️

① 종료 코드 처리 구멍

종료 코드 처리 로직에 구멍이 있어 예외 상황에서 처리 안 됨

💉

② ANSI 인젝션 위험

터미널 출력에 ANSI 이스케이프 문자 필터가 빠져 injection 위험

🔁

③ 경로 이중 적용 버그

경로를 두 번 적용하는 버그 — Claude 입장에서는 자기 코드라 못 봄

🚑 Codex Rescue 실제 사례: 사이드바 리팩터링

한 분이 사이드바 큰 리팩터링 후 Codex Rescue로 점검했더니 6분 35초 만에 엣지 케이스 버그 4개 발견:

🔇

조용한 제출 실패

설정 안 됐을 때 제출 버튼이 에러 없이 조용히 실패함

⏭️

리뷰 단계 강제 점프

지시문 없이 리뷰 단계로 건너뛸 수 있는 흐름 구멍

💀

뒤로가기 복구 불가

실행 중인 요청을 뒤로가기로 끊으면 복구가 안 됨

💥

📌 왜 이런 흐름이 생겼나 — 배경 스토리

처음엔 Claude Code 단독으로 쓰는 분들이 많았어요. 결과가 좋으니까요. 하지만 쓰다보면 불안한 순간들이 생겼고, 결정적으로 2~3월 성능 저하 사태를 겪으면서 분위기가 바뀌었습니다. Anthropic이 직접 사후 보고서를 내고 버그 3개를 인정하면서 "한 모델만 믿는 게 위험하다"는 감각이 퍼졌습니다.
