프롬프트만으로는 부족하다

프롬프트·컨텍스트 엔지니어링만으로는 부족하며, AI가 정보를 알아도 엉뚱한 행동을 하는 '규칙과 울타리'의 문제가 존재한다.

🔧

하네스 엔지니어링의 핵심 철학

AI 에이전트가 실수했을 때 프롬프트를 고치는 대신, 그 실패가 구조적으로 반복 불가능하도록 시스템 자체를 바꾸는 기술이다.

🏗️

4가지 기둥

①기계가 읽는 컨텍스트 파일 ②린터·구조적 테스트·pre-commit hook으로 규칙 자동 강제 ③도구 접근 권한 제한(물리적 차단) ④AI 생성 코드의 주기적 품질 점검

📈

개발자 역할의 상향 이동

코드 한 줄 작성에서 AI가 실수할 수 없는 환경을 설계하고 최종 책임을 지는 상위 레벨로 역할이 이동한다.

🔄 AI 활용 방법론 4가지 축

프롬프트 엔지니어링 — AI에게 말을 잘 거는 기술 기초

구체적인 요청으로 AI 출력 품질 향상, 그러나 천장이 존재

컨텍스트 엔지니어링 — 필요한 정보를 정확하게 제공 심화

프로젝트 구조, API 문서 등 필요한 것만 제공 — 많이가 아닌 정확히

하네스 엔지니어링 — AI가 실수할 수 없는 환경 설계 ★ 오늘의 주인공

규칙·울타리·테스트로 AI 행동을 시스템적으로 통제

에이전틱 엔지니어링 — 자율 워크플로우 설계 최상위

멀티 에이전트 조율, 추론 루프 설계 — 하네스와 함께 동작

"에이전트가 규칙을 어겼을 때, 더 잘해봐라고 프롬프트를 고치는 게 아닙니다. 그 실패가 구조적으로 반복 불가능하도록 Harness를 고치는 겁니다."

— 실밸개발자, 영상 본문 중

🐴 말과 마구(Harness) 비유

🏛️ 하네스 엔지니어링의 4가지 기둥

📄
기둥 1

기계가 읽는 컨텍스트 파일

claude.md, agents.md, cursor-rules 등은 단순 문서가 아닌 AI가 실행하는 런타임 설정 파일이다.

claude.md 설정 예시

  • • 새로운 라이브러리를 도입하지 마
  • • 기존 API 패턴을 무조건 따라야 해
  • • DB Query는 반드시 ORM을 통해서만 해야 해

→ 매번 프롬프트에 반복할 필요 없이 모든 작업에 자동 적용

🔍
기둥 2

린터·구조적 테스트·Pre-commit Hook

규칙을 문서에 써두는 것만으로 부족하다. AI는 규칙을 읽고도 어길 수 있기 때문에 시스템이 자동으로 강제해야 한다.

📝

린터

코드 맞춤법 검사기, 규칙 어기면 자동 에러

🧪

구조적 테스트

의존성 규칙, 폴더 간 import 제한 강제

🔗

Pre-commit Hook

저장 전 자동 검사, 빌드 실패 시 자동 수정

✨ 핵심: CI가 실패하면 에이전트가 스스로 수정 → 자동 교정 루프!

🚫
기둥 3

도구 접근 권한 제한 (물리적 차단)

프롬프트로 "절대 DB를 삭제하지 마라" → 부탁은 어길 수 있다.
도구 경계 = 물리적 차단 = 어길 수 없다.

파일 시스템

✓source 폴더 — 읽기/쓰기 가능
✗configuration 폴더 — 읽기만 가능

데이터베이스

✓SELECT — 가능
✗DROP TABLE — 절대 불가

API 호출

✓내부 API 호출 가능
✗외부 서비스 호출 불가

터미널 명령

✓Whitelist 명령만 허용
✗그 외 모든 명령 차단
♻️
기둥 4

AI 생성 코드의 주기적 품질 점검 (Garbage Collection)

AI 에이전트는 기존 코드의 나쁜 패턴을 그대로 복제한다. 마틴 파울러(Martin Fowler)가 명명한 자동 청소 시스템이 필수다.

→ 코딩 규칙 위반 자동 감지
→ 중복 코드 발견 및 제거
→ 리팩토링 PR 자동 생성
→ 안티패턴 주기적 체크

⚡ 진화적 특성: 에이전트가 실수할 때마다 새 규칙이 추가 → 시스템이 점점 더 견고해짐

하네스 시스템의 4단계 실행 흐름

📊 OpenAI 3명 팀 사례 분석

코드 한 줄 없이 5개월 만에 제품 배포한 비결

OpenAI 공식 블로그 발표 (2026년 2월) — 하네스 관점 분석

구성 요소 실제 한 일 하네스 관점
컨텍스트 파일 agents.md 작성 및 설계 AI에게 줄 지침서를 사전 설계 → 기둥 1
CI/CD 게이트 lint, test, hook 자동 파이프라인 구축 자동 검증 시스템 선구축 → 기둥 2
도구 경계 AI 에이전트 접근 범위·권한 사전 설정 물리적 접근 제한 → 기둥 3
피드백 루프 AI 코딩 → 리뷰 → 규칙 보강 반복 자동 품질 점검, 진화하는 하네스 → 기둥 4

⚖️ 프롬프트 고치기 vs 하네스 고치기

비교 항목 ❌ 프롬프트 수정 ✅ 하네스 수정
성격 부탁 (어길 수 있음) 강제 (어길 수 없음)
재발 가능성 높음 — 다음번에 또 실수 없음 — 구조적으로 불가
접근 방식 "DB를 직접 호출하지 마" 추가 아키텍처 테스트로 빌드 자동 실패
인간 개입 매번 필요 시스템이 자동 처리
시간에 따른 변화 계속 같은 실수 반복 시스템이 점점 견고해짐
