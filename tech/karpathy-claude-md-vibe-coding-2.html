🦠 카파시가 짚은 AI의 3대 고질병

🤔

고질병 ①

잘못된 가정

"로그인 기능 추가해줘" → 물어보지 않고 JWT 인증 시스템 200줄을 우르르 짬. 세션 쿠키 로그인 원했을 수도 있는데.

결과: 처음부터 가정이 틀려서 전부 다시 시작
📦

고질병 ②

코드 부풀리기

이메일 형식 확인 한 줄 → 옵션 8개짜리 거대한 검증 클래스, 설정 파일, 추상 인터페이스까지. "나중에 필요할 수도 있으니까"

결과: 3줄짜리가 200줄로. 읽기는 10배 어려워짐
🔧

고질병 ③

멋대로 고치기

한 함수만 수정 요청 → 옆 함수 포매팅 정리, 변수명 변경, 오래된 주석 삭제까지. 변경사항 50개로 폭발.

결과: 로그인 색깔 바꿨더니 회원가입 페이지가 먹통

📋 4원칙 상세 카드

🧠
고질병 ① 해결

원칙 1 — 코딩 전 먼저 생각하라

가정하지 마. 헷갈리면 먼저 질문하게 설정

  • → 자기 이해를 먼저 명확히 말하기
  • → 해석이 여럿이면 다 보여주고 고르기
  • → 더 단순한 방법 있으면 반박도 OK
  • → 모르면 멈추고 무엇이 헷갈린지 묻기
✂️
고질병 ② 해결

원칙 2 — 단순하게 먼저

문제를 해결하는 최소 코드, 추측은 없이

  • → 시키지 않은 기능 추가 금지
  • → 한 번만 쓸 코드 추상화 금지
  • → 일어날 수 없는 상황의 에러 처리 금지
  • → 200줄 → 50줄 가능하면 무조건 다시 짜기
🔬
고질병 ③ 해결

원칙 3 — 외과수술처럼 정확하게

꼭 수정해야 되는 것만 고쳐라

  • → 인접 코드·주석·포매팅 개선 금지
  • → 안 망가진 코드 리팩토링 금지
  • → 기존 스타일 그대로 따르기
  • → 모든 변경은 사용자 요청으로 소급 가능해야 함
🎯
카파시 최강조

원칙 4 — 목표 중심으로 실행

성공 기준을 정해. 검증될 때까지 반복해.

  • → "검증 추가" ❌ → "예상치 못한 입력 테스트 작성 후 통과" ✅
  • → "버그 수정" ❌ → "버그 재현 테스트 먼저, 통과시키기" ✅
  • → 여러 단계면 계획 + 각 단계별 검증 기준 먼저
  • → 강한 성공 기준 = AI 혼자 완주 가능

📊 명령 방식 Before / After 비교

상황 ❌ 이전 (모호한 명령) ✅ 이후 (성공 기준 제시)
테스트 "검증 추가해줘" "예상치 못한 입력에 대한 테스트 작성, 그다음 통과시키기"
버그 수정 "버그 수정해줘" "버그 재현하는 테스트 먼저, 그 다음 통과되게 수정"
리팩토링 "X 리팩토링해줘" "리팩토링 전후로 모든 테스트 통과 여부 확인"
로그인 "로그인 기능 추가해줘" "이메일+세션 쿠키 방식 로그인, JWT 불필요, 50줄 이내로"

🇰🇷 한국어 맞춤 커스텀 5가지

Rule 1 한국어 마침표

콜론으로 문장 끝내지 마. 마침표로 끝내. 한 줄 추가만으로 한국어 출력이 훨씬 자연스러워짐.

Rule 2 파일 헤더 주석

새 소스 파일 만들 때 맨 위에 한 줄 주석. AI가 전체를 읽지 않아도 파일 역할 파악 가능.

Rule 3 계획+체크리스트+컨텍스트

큰 작업 시작 전: 계획서 → 체크박스 체크리스트 → 의사결정 근거 컨텍스트 노트. 다음날 이어해도 물 흐르듯.

Rule 4 완료 전 테스트 실행

코드 수정 세션에서 "끝났습니다" 말하기 전 무조건 테스트 한 번. AI가 자꾸 이걸 빼먹어서 명시적으로 추가.

Rule 5 의미 단위 커밋

의미 있는 작업이 끝나면 즉시 커밋. "이 커밋 메시지를 한 문장으로 설명할 수 있는가?" — 가능하면 커밋, 불가능하면 분리.
